1. PRÉSENTATION GÉNÉRALE ET CONTEXTE STRATÉGIQUE 


1.1 Contexte et justification du projet 

YAOCOM'S GROUPE, acteur majeur de la distribution de matériel informatique et de 
solutions numériques en Côte d'Ivoire, fait face à des défis croissants de gestion 
intégrée de son activité : - Dispersion des outils : Utilisation d'outils bureautiques et de logiciels non 
interconnectés - Non-conformité fiscale partielle : Absence d'intégration native avec l'API DGI pour la 
Facture Normalisée Électronique (FNE) - Manque de traçabilité : Difficulté à retracer le cycle de vie complet d'une transaction - Décisions non assistées : Absence d'indicateurs prédictifs pour l'aide à la décision - Temps de traitement élevé : Processus manuels générant des délais et des erreurs 
Le présent projet vise à déployer WinTech ERP V2.5 Édition GIA, un système de gestion 
d'entreprise nouvelle génération qui automatise l'ensemble de la chaîne commerciale, 
logistique, RH et comptable, tout en garantissant une conformité fiscale totale avec la 
Direction Générale des Impôts (DGI) de Côte d'Ivoire. 

1.2 Vision stratégique 

WinTech ERP n'est pas un simple logiciel de saisie. C'est un Système d'Aide à la 
Décision (SAD) intelligent qui : - Transforme chaque transaction en donnée analytique exploitable - Automatise la cascade financière conformément aux standards Sage 100 Commercial - Garantit une conformité fiscale rigoureuse via l'intégration native DGI/FNE - Assure une traçabilité totale grâce à un Système d'Archivage Électronique (SAE) normé 
- Anticipe les besoins grâce à un assistant IA contextuel nommé GIA (Gestion Intégrée et 
Automatisée)

3. DESCRIPTION DÉTAILLÉE DES MODULES 
3.1 Module ATIC – Catalogue produits 
3.1.1 Fonctionnalités détaillées 
Le module ATIC constitue le référentiel central de tous les articles commercialisés. 
Chaque fiche article contient : 
Champs obligatoires : - Référence (génération automatique ou manuelle) - Désignation (libellé court) - Description technique (champ texte riche) - Catégorie (ex: Ordinateurs, Périphériques, Réseau) 
- Prix d'achat HT (en FCFA) - Photo HD (upload format JPG/PNG, max 5 Mo) - Unité de vente (pièce, lot, etc.) 
Champs calculés automatiquement : 
``` 
Prix de vente HT = Prix d'achat HT × 1,30 (marge fixe 30%) 
TVA = 18% (taux OHADA) 
Prix de vente TTC = PV HT × (1 + TVA/100) 
``` 
Génération de la référence intelligente : 
``` 
Format : REF-[INITIALES]-[AAAA]-[MM]-[JJ]-[HHMM]-[USER_ID] 
Exemple : REF-ORD-2025-04-13-1435-42 
``` 
3.1.2 Règles de gestion 
| Règle | Description | 
|-------|-------------| 
| RG-ATIC-01 | La marge de 30% est obligatoire et non modifiable | 
| RG-ATIC-02 | Un article ne peut être supprimé s'il apparaît dans un devis/facture | 
| RG-ATIC-03 | La photo produit est optionnelle mais recommandée | 
| RG-ATIC-04 | Le stock minimum est configurable par catégorie | 

3.2 Module CRM – Gestion clients 
3.2.1 Typologie des clients 
Client Particulier : - Nom, prénom - Email, téléphone, adresse - Ville, pays - Historique des achats 
Client Professionnel : - Raison sociale - IFU (Identifiant Fiscal Unique) – obligatoire pour facturation DGI - RCCM (Registre du Commerce et du Crédit Mobilier) - Logo de l'entreprise - Encours maximal autorisé - Contact principal 
3.2.2 Suivi de solvabilité 
Le système calcule automatiquement un indicateur de solvabilité basé sur : - Encours actuel / Encours maximal autorisé - Historique des paiements (retards, impayés) - Ancienneté de la relation commerciale 
| Seuil | Indicateur | Action recommandée | 
|-------|------------|--------------------| 
| < 70% | Vert | Vente normale | 
| 70% – 90% | Orange | Vente avec supervision | 
| > 90% | Rouge | Vente bloquée sauf validation manager | 
3.3 Module Commercial – Cascade financière Sage 100 
3.3.1 Algorithme de calcul 
La cascade financière reproduit fidèlement le moteur de calcul de Sage 100 
Commercial : 
``` 
Étape 1 : HT Brut 
= Σ (Quantité × Prix unitaire HT) 
Étape 2 : Remise Commerciale 
= HT Brut × (Remise % / 100) 
Net Commercial = HT Brut – Remise Commerciale 
Étape 3 : Escompte Financier 
= Net Commercial × (Escompte % / 100) 
Net Financier = Net Commercial – Escompte Financier 
Étape 4 : Frais de Port 
= Frais de port forfaitaires ou réels 
Net Taxable = Net Financier + Frais de Port 
Étape 5 : TVA (taux OHADA 18%) 
= Net Taxable × (18 / 100) 
Étape 6 : Net à Payer TTC 
= Net Taxable + TVA 
``` 
3.3.2 Interface de cotation 
L'interface de création de devis intègre : 
Recherche instantanée AJAX : - Auto-complétion clients dès la 3e lettre tapée - Auto-complétion articles avec affichage du prix et du stock - Debounce de 300ms pour éviter les appels serveur intempestifs 
Calcul dynamique temps réel : - Chaque saisie (quantité, remise, escompte) déclenche un recalcul complet - Affichage immédiat des totaux intermédiaires et final - Visualisation de la TVA collectée

3.5 Module Logistique 
3.5.1 Interface magasinier (Picking list) 
L'écran dédié au magasinier affiche : - Liste des factures certifiées en attente de préparation - Par défaut : tri par date d'émission croissante - Visualisation du détail des articles à prélever - Indication de l'emplacement de stockage 
Actions disponibles : 
1. Démarrer la préparation → Statut "En préparation" 
2. Valider la préparation → Statut "Prêt" (déstockage automatique) 
3. Signaler une rupture → Notification commercial via Ticker 
3.5.2 Gestion des ruptures de stock 
En cas de stock insuffisant : - Le magasinier clique sur "Rupture" pour l'article concerné - Le système crée une alerte dans le Ticker de tous les commerciaux - La facture reste en statut "En attente" - Notification push (email facultatif) envoyée au responsable approvisionnement 
3.6 Module RH et Comptabilité analytique 
3.6.1 Fiche employé et coût réel 
Chaque employé possède une fiche avec : 
| Champ | Description | Source | 
|-------|-------------|--------| 
| Salaire de base | Fixe mensuel | Saisie RH | 
| Primes | Indemnités, avantages | Saisie RH | 
| Charges patronales | CNPS, taxes diverses (15% forfaitaire) | Configuration | 
| Coût réel agent | Salaire brut + charges patronales | Calcul automatique | 
| Masse salariale | Σ coûts réels tous agents | Calcul automatique | 
3.6.2 Indicateurs clés (KPI) 
Le tableau de bord présente les indicateurs suivants : - Chiffre d'affaires HT (période courante vs période précédente) - Charges totales (achats, frais généraux, masse salariale) - Résultat net = CA HT – Charges - Marge brute = (CA HT – Coût d'achat) / CA HT × 100 - Rotation des stocks (jours) - Délai moyen d'encaissement (jours) 
3.7 Assistant IA GIA 
3.7.1 Capacités de l'assistant 
GIA (Gestion Intégrée et Automatisée) est un chatbot intelligent capable de : 
1. Interrogation du stock : - *Utilisateur :* "Combien de PC HP ProBook avons-nous en stock ?" - *GIA :* "Stock actuel : 23 unités (seuil d'alerte : 5). Dernière entrée le 10/04/2025." 
2. Assistance procédurale : - *Utilisateur :* "Comment créer un nouveau fournisseur ?" - *GIA :* "Navigation : Menu Configuration → Fournisseurs → Nouveau. Remplir IFU, 
RCCM, coordonnées bancaires. La validation est immédiate." 
3. Alertes prédictives : - *Alerte automatique :* "    
La marge sur la catégorie 'Ordinateurs' a baissé de 5% ce 
mois. Vérifiez les prix d'achat fournisseurs." 
4. Analyses avancées : - *Utilisateur :* "Quel est le client le plus rentable ce trimestre ?" - *GIA :* "SARL TECHNOLOGIE SERVICES – CA : 45M FCFA – Marge : 12.5M FCFA (28%)." 
3.7.2 Architecture technique de GIA 
Le chatbot repose sur : - Analyse lexicale légère (mots-clés prédéfinis) - Mapping requête → action métier (SQL ou documentation) - Pas d'IA générative externe (fonctionnement hors ligne) - Journalisation de toutes les interactions ---

4.3 Base de données – Détails techniques 
4.3.1 Colonnes générées (Generated Columns) 
MySQL 5.7+ supporte les colonnes virtuelles et stockées. WinTech ERP les utilise pour 
garantir 0% d'erreur comptable : 
```sql -- Colonne calculée : Prix de vente HT (marge 30%) 
pv_ht DECIMAL(10,2) GENERATED ALWAYS AS (pa_ht * 1.30) STORED -- Colonne calculée : Prix de vente TTC (TVA 18%) 
pv_ttc DECIMAL(10,2) GENERATED ALWAYS AS (pv_ht * 1.18) STORED -- Colonne calculée : Net à payer dans facture 
net_a_payer DECIMAL(15,2) GENERATED ALWAYS AS (net_taxable * 1.18) STORED 
```

7. SÉCURITÉ ET PROTECTION DES DONNÉES 
7.1 Politique de sécurité 
La sécurité est organisée selon 5 piliers : 
| Pilier | Mesures | 
|--------|---------| 
| Authentification | Bcrypt (12 rounds), session PHP sécurisée, expiration configurable | 
| Intégrité des données | Colonnes générées SQL, triggers, clés étrangères | 
| Confidentialité | .env hors dépôt, chiffrement des mots de passe, logs anonymisés | 
| Disponibilité | Backups quotidiens, mode dégradé DGI, redondance serveur | 
| Non-répudiation | Journal d'audit complet, signature électronique DGI | 
7.2 Protection contre les injections SQL 
Toutes les requêtes SQL utilisent des requêtes préparées via PDO : 
```php 
// Non sécurisé (interdit) 
$sql = "SELECT * FROM users WHERE email = '$email'"; 
// Sécurisé (obligatoire) 
$stmt = $db->prepare("SELECT * FROM users WHERE email = ?"); 
$stmt->execute([$email]); 
``` 
7.3 Protection CSRF 
Chaque formulaire POST inclut un token unique : 
```php 
// Génération 
$_SESSION['csrf_token'] = bin2hex(random_bytes(32)); 
// Vérification (middleware) 
if (!hash_equals($_SESSION['csrf_token'], $_POST['csrf_token'])) { 
throw new Exception('CSRF token invalide'); 
} 
``` 
7.4 Isolation des secrets 
Le fichier `.env` n'est jamais versionné (ajouté à `.gitignore`). Il contient : - Identifiants base de données - Clés API DGI - Secret key pour sessions - Configuration spécifique production ---

9. PROCESSUS DÉTAILLÉ DE DÉVELOPPEMENT 
9.1 Méthodologie 
Le développement suit une méthodologie Agile adaptée avec des sprints de 2 semaines. 
Chaque sprint livre un incrément fonctionnel testé et documenté. 
Cérémonies clés : 
- Planification de sprint (2h) – Définition des user stories - Daily meeting (15 min) – Synchronisation quotidienne - Revue de sprint (1h) – Démonstration au client - Rétrospective (1h) – Amélioration continue 
9.2 Rôles et responsabilités 
| Rôle | Personne | Responsabilités | 
|------|----------|-----------------| 
| Product Owner | Représentant YAOCOM'S | Définit les priorités, valide les livrables | 
| Scrum Master | Chef de projet technique | Anime les cérémonies, lève les blocages | 
| Lead Developer | Architecte technique | Valide l'architecture, code les parties critiques 
| 
| Développeur Full Stack | (2 personnes) | Développent les fonctionnalités | 
| Testeur QA | (1 personne) | Tests fonctionnels et non fonctionnels | 
9.3 Outils de développement 
| Catégorie | Outil | Usage | 
|-----------|-------|-------| 
| Versionnement | Git (GitHub) | Code source | 
| Gestion de projet | Jira / Trello | Suivi des sprints | 
| Documentation | Confluence | Spécifications | 
| Tests | PHPUnit | Tests unitaires | 
| Base de données | phpMyAdmin / Adminer | Administration | 
| IDE recommandé | VS Code / PHPStorm | Développement | 
9.4 Standards de codage 
- PHP : PSR-1, PSR-12 (normes d'écriture) - SQL : Indentation stricte, alias explicites - CSS : BEM (Block Element Modifier) - JavaScript : ES6+, modules, async/await - Commits Git : Convention Conventional Commits (`feat:`, `fix:`, `docs:`) ---
